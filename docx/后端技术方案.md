# 智慧旅游平台后端技术方案

## 一、项目概述

### 1.1 项目背景
智慧旅游平台是一个面向游客、商户、政府管理方的智能旅游服务系统，提供AI智能推荐、路线规划、地图打卡、内容分享、商户管理、政府监管等核心功能。

### 1.2 项目目标
- 为用户端小程序提供稳定的RESTful API服务
- 为商户端提供景点管理、内容发布、评论回复等功能
- 为政府端提供审核管理、数据统计、运营分析等功能
- 实现AI智能推荐和路线规划功能
- 支持多语言国际化

### 1.5 用户认证体系

本系统采用**三级用户认证体系**，平衡用户体验与账户安全：

#### 1.5.1 认证层级说明

| 层级 | 认证方式 | 可用功能 | Token有效期 |
|------|---------|---------|------------|
| 公共接口 | 无需认证 | POI搜索/浏览、官方内容浏览 | - |
| 匿名用户 | 自动匿名Token | 个性化推荐、AI规划(限次)、设置偏好 | 30天 |
| 正式用户 | 微信登录/手机号+密码 | 所有功能：打卡、评论、创建路线、消息等 | 7天 |

#### 1.5.2 注册类型枚举

| 值 | 说明 | 登录方式 |
|----|------|---------|
| 0 | 匿名注册 | 自动生成Token |
| 1 | 微信注册 | 微信授权code |
| 2 | 手机号注册 | 手机号+密码 |
| 3 | 微信+手机绑定 | 微信授权+手机号 |

#### 1.5.3 账号升级流程

```
匿名用户 ──(绑定手机号+验证码)──▶ 正式用户
   │                                  │
   │                                  ▼
   │                           可关联微信
   │                                  │
   ▼                                  ▼
┌─────────────────┐           ┌─────────────────┐
│ AI规划次数: 3次 │           │ AI规划次数: 5次 │
│ 功能受限        │           │ 全功能          │
└─────────────────┘           └─────────────────┘
```

#### 1.5.4 认证中间件设计

## 六、技术实现要点

### 6.1 多端认证机制

#### 6.1.1 用户端认证体系（JWT + 三级分层）

**公共接口（无需认证）：**
```
直接访问，不需要 Token
适用于：POI搜索、POI详情、内容浏览、热力图等
```

**匿名用户（optionalAuth）：**
```
1. 前端调用 POST /api/user/anonymous（无需任何参数）
2. 后端生成匿名 openid 和 JWT Token
3. Token 有效期 30 天
4. 匿名用户可使用：个性化推荐、AI规划(3次)、设置偏好
5. 匿名用户不可使用：打卡、评论、创建路线
```

**正式用户（requiredAuth）：**

微信登录流程：
```
1. 前端调用 wx.login() 获取 code
2. 后端通过 code 换取 openid
3. 查询用户是否存在，不存在则创建
4. 生成 JWT Token（7天有效期）
5. 返回 Token 和用户信息
```

手机号登录流程：
```
1. 前端调用 POST /api/user/phone-login
2. 验证手机号 + 密码（或验证码）
3. 查询/创建用户
4. 生成 JWT Token
```

**匿名用户升级为正式用户：**
```
1. 匿名用户调用 POST /api/user/send-code 发送验证码
2. 匿名用户调用 POST /api/user/bind-phone 绑定手机
3. 验证验证码，更新 User.tel 和 register_type
4. 返回新 Token，标记为正式用户
5. AI规划次数从3次提升到5次
```

**手机用户关联微信：**
```
1. 手机用户调用微信登录
2. 如果 openid 对应的微信账号已存在
3. 系统检测到手机号冲突，触发账号合并
4. 合并两个账号的数据
5. 返回合并后的用户信息
```

#### 6.1.2 商户端（Session + JWT）：

```
1. 商户端使用手机号+密码登录
2. 后端验证后生成 JWT Token
3. 返回 Token 和商户信息
```

#### 6.1.3 政府端（Session + JWT）：

```
1. 政府管理员使用用户名+密码登录
2. 后端验证后生成 JWT Token
3. 根据 role 控制权限
```

### 6.2 AI路线生成流程

```
1. 前端提交用户需求（文字描述）
2. 后端使用 NLP 解析用户输入
3. 根据偏好从 poi_tag_rel 表筛选候选 POI
4. 将候选 POI 和用户画像提交给 LLM
5. LLM 返回结构化路线数据
6. 保存路线到 routes_info 表
7. 返回路线 ID 和详情
```

### 6.3 多语言支持

数据库中多语言字段采用 JSON 格式存储：
```json
{
  "zh": "景点名称",
  "en": "Attraction Name"
}
```

服务层根据请求头 `Accept-Language` 返回对应语言内容。

### 6.4 安全措施

- JWT Token 认证，有效期 7 天
- 密码使用 bcrypt 加密存储
- SQL 注入防护（Prisma参数化查询）
- XSS 防护（输入过滤）
- CORS 跨域配置
- 商户端与政府端接口分开认证

## 九、开发规范

### 9.1 代码风格
- 使用 JavaScript ES6+
- 遵循 ESLint 规范
- 使用 async/await 处理异步
- 统一错误处理

### 9.2 命名规范
- 控制器/路由：`*Routes` 后缀
- 服务：`*Service` 后缀
- 模型：`*Model` 或 Sequelize模型名

### 9.3 接口前缀规范

| 端口 | 前缀 | 示例 |
|------|------|------|
| 用户端 | `/api/` | `/api/user/login` |
| 商户端 | `/api/merchant/` | `/api/merchant/poi` |
| 政府端 | `/api/gov/` | `/api/gov/poi/audit` |

### 9.4 Git 提交规范
```
feat: 新功能
fix: 修复bug
docs: 文档更新
style: 代码格式
refactor: 重构
test: 测试
```
